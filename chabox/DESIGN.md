# ChaBox Design Philosophy & Architecture Guide (DESIGN.md)

ChaBox follows a **"Scope-Driven"** design pattern. We organize the application into distinct functional scopes (`page/<scope>`). Each scope acts as an autonomous unit, naturally embodying the MVC (Model-View-Controller) pattern through its directory structure rather than through rigid, forced naming conventions.

## 1. Core Architectural Philosophy

*   **Domain Cohesion**: Every `page/<scope>` directory is a self-contained feature module. It contains not just the UI, but also the business logic specific to that domain.
    *   **Controller (C)**: The files at the root of `page/<scope>/`. These serve as the **navigation entry points**. They coordinate business flows, initialize states, and act as the bridge between the View and the Model.
    *   **View (V)**: Composed of `layout/` and `widget/`. `layout` defines the page structure, while `widget` defines the atomic UI components that make up the page.
    *   **Model/Business Logic (M)**: Data processing logic. This can be global, shared utilities in `utils/` or domain-specific business helpers in `helper/`.

*   **Flexible Refactoring**: We avoid over-engineering directory structures. When a `build` method grows too large:
    *   **Preferred Approach**: Simply split the code into private methods (e.g., `_buildSection()`) within the same file—it’s simple and clean.
    *   **Advanced Approach**: Extract the logic into an independent `widget/` file only when the UI component needs to be reused or the logic is too complex to remain in the main file.

*   **General vs. Specific Boundaries**:
    *   `lib/widget/`: Atomic components shared across the entire project.
    *   `page/<scope>/widget/`: Atomic components exclusive to that specific scope.

## 2. Architectural Action Rules

*   **Business Logic Decision Model**:
    *   **Stay in Scope (`helper/`)**: If the logic handles domain-specific "business workflows."
    *   **Promote to Project Level (`utils/`)**: If the logic provides generic "technical capabilities" (e.g., encryption wrappers, config loaders).
*   **Data Flow Contract**:
    *   `C` (Controller) acts as the data hub, listening for changes in the Model.
    *   `V` (Atomic UI) should receive data via constructor parameters (Props) and bubble up events via callbacks. UI components should not mutate global states directly; they should report actions to the `C` layer.
*   **Routing as Entry Point**: All navigation routes target the root controller file of a `Scope`. This is the standard "North Star" for anyone looking to find where a feature begins.

## 4. Quality & Maintainability

We uphold a high standard for code quality to ensure long-term maintainability:

1. **High-Quality Documentation (English Comments)**: All business methods under `helper/` and `Public APIs` for each feature module must contain English comments. Comments should explain "why" (design intent/trade-offs) rather than repeating what the code does.
   * *Example*: `// We use a debounce here to prevent excessive disk I/O when saving notes during typing.`
2. **Accountability**: Every developer is the ultimate owner of their contributed code. You are expected to:
   * Conduct thorough code logic review.
   * Perform edge-case testing.
   * Verify copyright compliance (ensure no restricted third-party code).

## 5. AI-Assisted Coding

We welcome the use of AI tools to assist in development, provided they adhere to our architectural contract:

1. **Architectural Consistency**: Any code generated by AI must strictly adhere to the ChaBox "Scope-Driven" architecture. It is strictly forbidden for AI to scatter business logic across global levels or call across scopes directly.
2. **AI Tagging**: Developers must tag the code source in PRs or commit messages (e.g., `[AI Generated]` or `[AI Assisted]`) and certify that the code has been reviewed according to the Quality & Maintainability standards defined in Section 4.

## 6. Defensive Architectural Principles

ChaCrypt's architectural design aims to improve the security level of data processing through the synergy of multiple defensive measures. We employ a multi-layered defense strategy, seeking to increase logical processing complexity and defensive depth, thereby raising the barrier for unauthorized functional cloning and reverse engineering.

#### 6.1 Deep Defense Philosophy: Strengthening Boundaries and Reconstructing Trust
ChaCrypt adopts a "defensive penetration" strategy, attempting to integrate security judgment logic into business processing flows. By intertwining security discernment with data processing workflows, we aim to enhance the system's logical interference resistance.
*   **Functional Defense against AI Models**: Through Semantic Masking and Logic Interference, the system attempts to mask judgment mechanisms as routine business processes like data alignment or buffer normalization. Such interference designs aim to weaken the effectiveness of feature extraction by large language models on system behavioral logic, thereby increasing the difficulty of generating effective functional clones.
*   **Defense Hardening at the Binary Level**: Addressing traditional binary reverse analysis, the system introduces Endogenous Integrity Validation. By constructing runtime integrity links, we attempt to impose constraints on binary instruction tampering and unauthorized interception of critical execution paths, aiming to reduce the risk of sensitive data leakage or illegal decryption.

#### 6.2 Full Lifecycle Admission Control: Behavior as a Gateway
The system tends to employ dynamic admission mechanisms, delegating the judgment authority for sensitive interactions (e.g., encryption/decryption, archiving, wiping) to closures dynamically injected at runtime.
*   **Adversarial Challenge**: Validation logic is constructed dynamically at runtime, attempting to break reliance on static compile-time structures. This design aims to increase the difficulty for attackers to directly infer complete execution chains from pre-compiled code.
*   **Assistance in Tamper Resistance**: The validation chain based on the state container (ProcessedResult) enforces that data must satisfy specific validation rules (e.g., feature alignment, length parity) along its processing path. This validation link aims to build an end-to-end data consistency defense line, thereby increasing the detection difficulty for malicious data hijacking.

#### 6.3 User Data Security: The Original Intention of the Defense System
ChaCrypt's defensive guidelines are committed to maximizing user data security protection.
*   **Enhancing Data Logic Anti-Reverse Strength**: Through the deployment of defensive architecture, we aim to impart stronger logical resistance to data during storage and processing, thereby increasing the complexity for unauthorized static analysis programs to forcibly understand data characteristics without comprehending the workflow.
*   **Building a Trusted Execution Defensive Posture**: This architecture strives to provide users with a data processing space equipped with certain logical barriers, aiming to reduce the potential attack surface data faces throughout its lifecycle from an architectural perspective, providing more rigorous logical protection for users' data assets.
