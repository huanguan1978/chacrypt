# L10N Naming Convention (多语言命名规范)

### 1. 为什么要这份规范？
随着项目迭代，多语言 Key 会迅速膨胀。这份规范旨在通过**“路径化”**命名，实现以下目标：
*   **消除歧义**：通过四级结构，精准定位 UI 元素在 App 中的位置。
*   **IDE 友好**：利用自动补全功能，大幅减少手动拼写错误。
*   **语义解耦**：将文案的“位置(Area)”与“功能(Key)”分离，重构 UI 时改动成本最小化。
*   **团队协作**：建立统一的语感，无需查阅文档即可猜出 Key 的命名。

---

### 2. 核心范式：`{Module}_{Container}_{Area}_{Key}`
每一级通过 `_` 分隔。为了保证路径清晰，**前三级严禁使用下划线**。

| 层级 | 说明 | 规则 | 示例 |
| :--- | :--- | :--- | :--- |
| **Module** | 业务大模块 | 单单词/小驼峰 | `auth`, `userProfile`, `common` |
| **Container** | 页面或容器 | 单单词/小驼峰 | `login`, `index`, `dialog` |
| **Area** | 逻辑区域 | 单单词/小驼峰 | `header`, `form`, `footer` |
| **Key** | 功能与后缀 | 语义描述 + 后缀 | `submit_action`, `welcome_label` |

---

### 3. 后缀标准 (Suffix Dictionary)
Key 的末尾必须根据文案属性添加后缀，用于描述其 UI 表现：
*   `_label`：静态文本、说明文字。
*   `_action`：按钮、可点击图标、交互动作。
*   `_hint`：输入框占位符 (Placeholder)。
*   `_error`：校验失败后的红色提示文案。
*   `_title`：页面标题、弹窗/卡片头部标题。

---

### 4. 最佳实践与补丁建议 (Pro Tips)
为提高开发效率，请遵循以下进阶规则：
*   **处理复合单词**：若模块名或页面名由多个单词组成，**严禁使用下划线连接**，请使用**小驼峰命名法**（例如：`userProfile` 代替 `user_profile`）。这能确保下划线始终作为层级的“唯一分隔符”。
*   **动态参数约定**：当文案包含动态变量时（如 `{username}`），命名时应忽略参数，仅描述该位置的**业务含义**（例如：`home_index_header_welcome_label`，内容为 `Welcome {username}`）。
*   **动作统一性**：若同一功能出现在不同位置，尽可能复用 `Key`。例如，`common_global_action_save_action` 可以被全站各处的保存按钮调用。

---

### 5. 禁止事项 (Anti-Patterns)
1.  **禁止模糊命名**：严禁使用 `ok`, `btn1`, `text_a` 等无意义命名。
2.  **严禁下划线滥用**：前三级命名中出现下划线将被视为严重违规。
3.  **禁止 Key 后缀缺失**：即便文案极其简单，也必须标明类型，以防 UI 样式调整（如将 `Label` 改为 `Button`）时产生歧义。
4.  **禁止硬编码**：禁止在 Dart 代码中直接写死字符串，无论文案多短，一律使用此规范对应的 Key。

---

### 6. 执行检查表 (Developer Checklist)
在提交 PR 之前，请确认你的 Key 是否符合：
- [ ] **层级验证**：是否由 4 个部分组成？
- [ ] **符号验证**：前三级是否含有下划线？（若有，请使用驼峰法改为单单词）
- [ ] **后缀验证**：Key 是否包含了规范后缀（如 `_action`, `_label`）？
- [ ] **语义验证**：文案功能是否清晰，是否可被其他同类页面复用？

---

> **核心心法**：看到 `_` 即进入下一层；只要是前三级命名，就绝不允许命名内出现第二个 `_`。这套方案将使你的多语言文件从“混乱堆砌”进化为“结构化数据库”。
